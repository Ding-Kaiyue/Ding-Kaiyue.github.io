从硬编码到插件化（上篇）：脏架构的问题诊断

发表于 2025-11-01 分类于 控制架构篇-从模式到工厂 本文字数： 7.1k 阅读时长 ≈ 6 分钟
本文记录我在重构机器人控制系统时遇到的“脏架构”问题，分析它为何难以扩展、难以维护，为下篇的清洁设计铺垫基础。

从硬编码到插件化：我如何重构控制器注册机制

这是一篇关于机器人控制系统架构设计的深度分析，揭示为什么简单堆积代码的方式会导致系统变得越来越脆弱。

本文适合：架构设计师、ROS2 开发者、以及关心代码质量的工程师

内容：从真实的混乱代码开始，分析 8 个严重的架构问题，然后通过 SOLID 原则揭示这些问题的根本原因。

开篇 - 一个现实的困境

想象一下，你正在开发一个机器人控制系统。系统需要支持多种运动模式：

  • 关节速度控制（JointVelocity）
  • 笛卡尔空间速度控制（CartesianVelocity）
  • 关节空间插值（MoveJ）
  • 直线运动（MoveL）
  • 圆弧运动（MoveC）
  • 轨迹回放（TrajectoryReplay）
  • …

每个模式都是一个完整的控制算法，需要独立的消息处理、参数管理、状态维护。随着功能的增加，系统变得越来越复杂。

问题来了：当你需要添加新的运动模式时，你要改动哪些地方？

对我来说，这个问题的答案一开始非常”直接”——我在主控制节点（StateMachine）里添加新的回调、订阅新的话题、修改状态机逻辑……一切都在一个文件里完成。

它”能跑”，甚至能运行得相当不错。但随着模式数量增加到 10、15、20 个，问题开始浮现。

每次添加一个新的控制模式，我都要打开那个越来越膨胀的文件。光是找到要改的函数，就要滚动好半天。改完了一个 case，又得祈祷不会打断别的模式。两个人并行开发两个新模式？那就准备好解决 merge 冲突吧。

系统变得越来越臃肿、越来越脆弱。修改一个地方要担心打破另一个地方。这，就是我后来称之为的 “脏架构（Dirty Architecture）”——不是说代码质量差，而是说职责混乱、耦合严重、无法优雅扩展。

所谓脏架构，不是指代码写得丑，而是指系统的结构无法支撑长期演化。它能工作，但无法成长。

接下来，我们从那段早期的真实代码开始，看看它到底”脏”在哪里——以及为什么会变成这样。

快速对比：从脏到清洁的演变

在深入问题之前，让我们先看一个”目标”——这样你能理解我们要解决什么问题：

架构维度 脏架构（Before） 清洁架构（After）
控制器注册 手写 if-else / switch-case 动态注册（工厂 + YAML）
控制器创建 new + 手动管理 工厂方法自动创建
添加新模式 修改核心文件 6-8 处 添加独立类文件 + 改 1 行 registry
通信接口 直接 ROS 订阅混在逻辑里 统一的抽象接口（ModeControllerBase）
配置方式 硬编码在代码中 YAML 配置驱动
核心代码变化 随功能数量线性增长 保持稳定（与功能数无关）
开发周期 1-3 天/控制器 2-4 小时/控制器
可测试性 无法单独测试 控制器独立可测

看这个对比表，你会发现：脏架构的核心问题是添加新功能要改动核心代码；清洁架构的核心优势是添加新功能无需改动核心代码。

这就是我们在下篇要实现的目标。

1.2 这个架构暴露的 8 个严重问题

这 8 个问题看似独立，但其实可以分成三大类。让我一个个讲解：

🔗 第一类：架构复杂度问题（职责混乱、耦合度高）

问题 1：订阅地狱

构造函数中硬编码了 10+ 个订阅，修改话题需要改代码并重新编译，无法灵活配置。

我一开始觉得这样写最直接——所有订阅都在一个地方。但当我添加第 5 个模式时，我意识到这个构造函数已经成了”意大利面”。

问题 2：巨大的 switch-case

单个函数 > 500 行，修改一个模式可能影响其他模式，merge 冲突频繁。

我每次改 switch-case 都要小心翼翼。改了一个 case，又得担心另一个 case 是否会受影响。到了第 10 个模式，这个函数已经大到我根本不敢重构。

问题 5：全局状态变量爆炸

20+ 个成员变量管理各种状态，竞态条件风险高，难以跟踪状态变化。

每添加一个新模式，我就要添加 5-10 个新的成员变量。什么 cartesian_thread_running_，stop_cartesian_thread_，active_twist_direction_ 这种。到最后，我甚至不知道这些变量是干什么的。

🔌 第二类：可扩展性问题（无法配置、代码重复）

问题 3：消息类型混乱

每个模式用不同的消息类型，没有统一的接口，容易发送错误的消息。

JointVelocity 订阅的是 JointState，CartesianVelocity 订阅的是 TwistStamped，MoveJ 订阅的是 Float64MultiArray。完全没有规律。新来的人要添加一个控制器，还要去翻之前的代码猜猜”应该用哪个消息类型”。

问题 6：无法动态配置

所有参数硬编码在代码中，部署不同的机器人需要修改代码。

话题名死死地写在代码里。想在另一个机器人上跑同样的代码？改话题名，重新编译。想根据不同的配置启用/禁用某个模式？没办法，代码里已经死了。

问题 7：每个模式自己实现线程管理

代码重复，线程频率不一致，难以统一管理生命周期。

CartesianVelocity 用 rclcpp::Rate(50) 循环，JointVelocity 用 std::this_thread::sleep_for(100ms)。没人规定应该怎样做。复制粘贴就完事了。结果：线程泄漏、死锁、同步问题……

🧪 第三类：可维护性问题（职责混乱、难以测试）

问题 4：业务逻辑与基础设施混合

业务逻辑、线程管理、ROS I/O 混在一起，无法独立测试。

想单独测试 CartesianVelocityController 的 IK 计算？不行，它和线程、锁、状态变量都混在一起。测试它就意味着要启动整个 StateMachine、初始化所有 ROS 通信。单元测试变成了”集成测试”。

问题 8：难以测试和扩展

无法独立单元测试某个控制器，耦合太严重。

两个同事要并行开发两个新模式？10 分钟后就开始出现 merge 冲突。因为他们改的是同一个文件的不同部分——构造函数、switch-case、成员变量。一个人改了，另一个人的代码就可能和新的部分冲突。

1.3 问题的根本原因

这 8 个问题的根源不在实现细节，而在架构设计的两个核心缺陷：

缺陷 A：中心化的”万能管理器”

所有控制逻辑、消息处理、线程管理都集中在 StateMachine 一个类中。任何改动都影响整个系统。

缺陷 B：硬编码而非配置驱动

添加新模式需要修改代码的多个位置，任何配置改动都需要重新编译。

关键洞察：这不是写法问题，而是架构设计的问题。脏架构的本质缺陷是没有清晰的职责边界。

脏架构的本质：

  • ❌ 能工作（短期）✓
  • ❌ 无法成长（长期）✗
  • ❌ 无法预测（修改一处，不知道会否影响另一处）

我们期待的理想状态（清洁架构）：

  • ✓ 添加新控制器，无需修改核心代码
  • ✓ 配置文件能驱动系统行为
  • ✓ 通信逻辑与控制逻辑完全解耦
  • ✓ 每个控制器可独立测试

这，就是我们要在下篇实现的目标。

第一部分（续）：8 个问题速览

用一张表快速总结脏架构的全貌：

序号 问题名 表现 根本原因
1️⃣ 订阅地狱 构造函数 10+ 个硬编码订阅，修改话题需改代码重编译 没有配置文件支持
2️⃣ 巨大 switch-case 500+ 行的单一函数，逻辑混乱，merge 冲突频繁 没有工厂模式分离逻辑
3️⃣ 消息类型混乱 每个模式用不同消息类型，无法统一接口 没有基类接口约束
4️⃣ 逻辑与基础设施混合 业务逻辑 + 线程管理 + ROS I/O 混在一起 没有职责分离（SRP）
5️⃣ 状态变量爆炸 20+ 个成员变量管理各种状态，竞态条件风险高 没有状态封装边界
6️⃣ 无法动态配置 所有参数硬编码，无法参数化部署 没有配置驱动设计
7️⃣ 线程管理重复 每个模式自己实现线程循环，代码重复高 没有统一的线程抽象
8️⃣ 难以测试和扩展 无法独立单元测试某个控制器 耦合太严重，无法隔离

一句话总结：所有问题都源于同一个根因——职责没有清晰边界，所有东西混在一个文件里。

第一部分（续）：脏架构的症状总结

随着控制器数量增加，脏架构的代价指数级增长：

控制器数量 核心文件行数 每个控制器成本 维护难度 状态
1-5 个 300-800 行 2-4 小时 ⭐ 简单 ✓ 可接受
5-10 个 800-1500 行 4-8 小时 ⭐⭐⭐ 中等 ✓ 还能接受
10-20 个 1500-3000 行 8-24 小时 ⭐⭐⭐⭐⭐ 困难 ⚠️ 需要重构
20+ 个 3000+ 行 24-72 小时 ⭐⭐⭐⭐⭐⭐ 极难 ❌ 不可维护

关键观察：

  • 核心文件行数呈线性增长（每个新控制器添加 150-200 行）
  • 添加新控制器的成本呈指数增长（从 2h → 4h → 8h → 24h）
  • merge 冲突从偶发 → 频繁 → 每次都有

当你超过 20 个控制器时，整个系统已经基本不可维护。

第二部分 - 底层理论 - SOLID 原则在本项目中的应用

清洁设计之所以有效，是因为它遵循了 SOLID 设计原则。理解这些原则能帮助我们看清脏架构为什么出现这些问题。

S - 单一职责原则（Single Responsibility Principle）

原则：一个类应该只有一个改变的理由。

脏架构的违反：

  • StateMachine 类同时负责：ROS 通信、模式切换、业务逻辑、线程管理、状态维护
  • 修改任何一个方面都要改这个类

收益：

  • 修改某个控制器的逻辑，不影响 ControllerManager
  • 代码复用性高，易于测试

O - 开闭原则（Open/Closed Principle）

原则：软件应该对扩展开放，对修改关闭。

脏架构的违反：

  • 添加新模式 → 必须修改 StateMachine 类
  • 每次扩展都要改现有代码

收益：

L - 里氏替换原则（Liskov Substitution Principle）

原则：子类应该可以替代父类，而不会导致程序出错。

脏架构的违反：

  • 没有统一的接口，每个控制器实现都不同

收益：

  • 可以轻松替换控制器实现
  • ControllerManager 代码永不修改

I - 接口隔离原则（Interface Segregation Principle）

原则：客户端不应该被迫依赖它不使用的接口。

脏架构的违反：

  • StateMachine 依赖 ROS、Kinematics、HardwareManager 等所有系统
  • 修改任何依赖都要改 StateMachine

收益：

  • 修改 Kinematics → 只影响 CartesianVelocityController
  • ControllerManager 完全不受影响

D - 依赖倒置原则（Dependency Inversion Principle）

原则：高层模块不应该依赖低层模块，两者都应该依赖抽象。

SOLID 原则对比表

原则 脏架构问题 清洁设计应用 收益
S 单一职责 StateMachine 负责所有 每个类只做一件事 修改隔离、易于测试
O 开闭 扩展=修改核心文件 扩展=添加新文件 风险低、成本恒定

D 依赖倒置 高层依赖低层 都依赖抽象 松耦合、易扩展

为什么 SOLID 原则很重要

SOLID 原则不是”最佳实践”的意见，而是经过几十年验证的工程原则。当你遵循这些原则时：

  1. 代码变得可持续演化 - 添加功能成本恒定
  2. 团队可以并行工作 - 不会频繁冲突
  3. bug 修复风险低 - 修改隔离在小范围内
  4. 新成员易上手 - 模块化的代码更容易理解
  5. 测试变得可行 - 解耦的代码易于单元测试

这些不仅仅是”代码美观”的问题，而是影响整个项目生命周期成本的工程决策。

一句话总结：

如果你想让系统能不断添加控制器而不触碰旧代码，就必须遵守 SOLID——特别是开闭原则和依赖倒置原则。

总结：问题诊断完成

现在我们清楚地看到了脏架构的 8 个严重问题，以及它们的根本原因——职责混乱和过度耦合。

这些问题都违反了 SOLID 原则，特别是：

  • 单一职责原则：StateMachine 职责太多
  • 开闭原则：每次扩展都要修改核心代码
  • 接口隔离原则：所有依赖混在一起
  • 依赖倒置原则：高层直接依赖低层实现

下一步：清洁设计的解决方案

在下篇中，我们将看到如何通过：

  1. 分离职责 - 每个控制器独立成类
  2. 配置驱动 - YAML 文件而非硬编码
  3. 工厂模式 - 动态创建控制器
  4. 统一接口 - 所有控制器实现同一个接口
