控制器体系的演进（中篇）：四层继承体系的设计与决策

发表于 2025-11-06 分类于 控制架构篇-从模式到工厂 本文字数： 9.3k 阅读时长 ≈ 8 分钟
本文讲解四层继承体系的核心设计，分析每一层的职责边界、模板使用的边界及其设计权衡。

接上篇《控制器体系的演进（上篇）：为什么需要分层继承》，本篇重点讲解四层体系的核心设计 — 每一层的职责边界、为什么采用这样的分层方式、以及这些决策背后的权衡。

本文适合：想要深入理解分层设计原理的C++开发者、架构师

核心观点：

  • 四层继承体系中，模板是为了处理类型多样性而存在的
  • VelocityControllerBase 和 TrajectoryControllerBase 需要模板处理不同的消息类型
  • TeachControllerBase 采用固定类型设计，避免不必要的模板复杂性
  • UtilityControllerBase 不处理消息，采用事件驱动的设计方式
  • 这四种设计方案分别对应了四种不同的控制交互模式

1. 四层继承体系详解：从接口到模板实现

理解了分层的必要性后，我们需要解决一个更具体的问题：每一层应该承载什么职责？模板应该出现在哪一层？下面以 VelocityController 为例，逐层剖析四层继承的设计细节。

四层的继承关系如下：

1.1 第一层：ModeControllerBase的统一基础接口

职责：定义所有控制器的基础接口和生命周期

设计要点：

  • 使用 std::any 作为消息的通用容器，允许任何消息类型
  • handle_message() 是虚函数，由第二层基类具体实现
  • 每个方法都需要 mapping 参数，支持多个机械臂独立控制

1.2 第二层：VelocityControllerBase（纯虚基类）

VelocityControllerBase 是第二层，定义了所有速度控制器的统一接口。它继承自 ModeControllerBase，但不直接实现具体的类型转换逻辑。

为什么需要第二层纯虚基类？

速度控制器需要处理多种不同的消息类型（JointState、TwistStamped 等），但都遵循相同的控制逻辑。第二层提供了一个统一的接口，让不同的具体控制器共享基础设施。

1.3 第三层：VelocityControllerImpl<T>（模板实现类）

第三层是模板实现类，自动处理消息类型的转换和映射（mapping）管理。关键职责：

  • 编译期类型检查：handle_message() 负责 std::any 到具体类型 T 的转换
  • 统一的映射管理：通过 get_mapping_from_message() 从消息中提取mapping，支持多机械臂
  • 子类简化：只需实现 velocity_callback(mapping, typed_msg) 处理业务逻辑

  • 指定消息类型：VelocityControllerImpl<sensor_msgs::msg::JointState>
  • 无需再实现类型转换逻辑，第三层已经处理
  • 只需关注业务逻辑：如何处理接收到的速度指令

1.5 其他控制模式的层次结构（Trajectory / Teach / Utility）

虽然我们以 VelocityController 详细展示了四层结构，但其他控制模式遵循类似的模式。这里简要说明它们的特点。

1.5.1 TrajectoryControllerBase 和 TrajectoryControllerImpl<T>

TrajectoryController 的四层结构与 VelocityController 完全相同，只是处理的消息类型和控制逻辑不同。

为什么需要模板？

因为轨迹控制器也处理多种不同的消息类型：

  • MoveJController 处理 sensor_msgs::msg::JointState（目标关节位置）
  • MoveLController 处理 geometry_msgs::msg::Pose（目标笛卡尔位置）
  • MoveCController 处理geometry_msgs::msg::PoseArray（目标笛卡尔位置序列）

模板同样用于编译期类型安全。

1.5.2 TeachControllerBase（特殊情况：不需要模板）

TeachController 的设计与前两者不同。所有记录控制器都使用相同的消息类型：std_msgs::msg::String（点/轨迹名称）。不存在多种消息类型的需求，因此无需模板。

1.5.3 UtilityControllerBase（特殊情况：不需要消息处理）

UtilityController 是最特殊的一层。工具类控制器不依赖ROS消息驱动，而是通过其他机制（ROS2 Action、内部状态机、定时器）驱动。

2. 核心设计决策：模板的选择与边界

四层继承的结构已经清晰，但三个关键决策仍需解释：为什么VelocityController需要模板而TeachController不需要？为什么UtilityController的handle_message是空实现？这些决策背后的逻辑，决定了体系的扩展性。

2.1 类型安全权衡：为何Velocity和Trajectory需要模板

问题：模板会增加代码复杂度，为什么还要用？

答案：类型安全 vs 代码简洁的权衡。

方案A：无模板（运行时检查）：

问题：

  • 每个控制器都要写 std::any_cast
  • 容易写错类型（编译期无法检查）
  • 如果 cast 失败，运行时才发现

方案B：模板化（编译期检查）：

优势：

  • ✅ 编译期就检查出类型错误
  • ✅ 消除代码重复（cast 逻辑在模板中）
  • ✅ 统一的异常处理
  • ✅ 子类无需关心 std::any，直接处理具体类型

2.2 消息类型多样性：为何Teach不需要模板

问题：为什么TeachControllerBase和VelocityControllerBase的处理方式不同？

答案：消息类型的多样性决定了设计方案。

  • VelocityControllerBase：子类使用多种消息类型

    • JointVelocityController → JointState
    • CartesianVelocityController → TwistStamped

    → 需要模板处理多种类型

  • TeachControllerBase：所有子类都使用同一种消息类型

    • PointRecordController → String
    • TrajectoryRecordController → String
    • PointReplayController → String
    • TrajectoryReplayController → String

    → 不需要模板，直接固定类型

这体现了SOLID的单一职责原则：

  • 如果需要处理多种类型，用模板承担这个职责
  • 如果消息类型固定，直接实现，不引入不必要的复杂度

2.3 驱动方式差异：为何Utility无需消息处理

问题：为什么UtilityControllerBase不处理消息？

答案：这类控制器采用不同的驱动方式。

前三种（Velocity、Trajectory、Teach）都由ROS话题消息驱动，而Utility不同：

  • UtilityControllerBase：不依赖ROS消息，而是通过其他机制驱动
    • ROS2ActionControlController → 由Action Server驱动
    • Move2InitialController → 由内部状态机驱动
    • Disable → 进入模式后立即触发

因此handle_message()是空实现，而start()是唯一的入口。
